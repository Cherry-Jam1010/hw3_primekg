# PrimeKG 精神健康项目前端设计修改说明

## 1. 当前版本的核心问题

当前前端虽然已经具备了基础功能，包括：
- 本地离线展示
- 中文化界面
- 基于实验产物的核查交互
- 语料库筛选与回填
- 按 `plan.md` 组织的三幕式结构

但从展示效果来看，当前版本仍然明显偏向“课程作业展示页”，而不是“有冲击力、可操作、可沉浸体验的交互作品”。主要问题如下：

### 1.1 视觉气质不对

- 页面更像信息看板，而不是“茧房被击穿”的叙事体验。
- 卡片、表格、指标的默认排布太规整，缺乏情绪张力。
- 配色偏稳妥，缺少从“封闭”到“击穿”的视觉转场。
- 没有形成强记忆点，用户看完后只会觉得“信息很多”，不会觉得“被震到”。

### 1.2 交互操作性不强

- 现在的主要动作仍然是“输入一句话 -> 点击按钮 -> 看结果”。
- 缺少探索感、反馈感、游戏化感。
- 用户不会主动想多玩几次，交互缺少诱因。

### 1.3 动画与节奏不足

- 当前页面更像静态内容拼接。
- 没有明显的分幕推进感。
- “检索”“校验”“证据浮现”“茧房裂开”这些关键动作没有被动画化。
- 页面没有形成“先压抑、再击穿、再看清结构”的动态节奏。

### 1.4 信息表达方式仍然偏“解释型”

- 虽然已经删减了一部分图表，但整体仍然容易落回“说明文字很多”的问题。
- 关键结果应该更多依赖视觉结构、状态变化、空间层次来表达，而不是一大段说明。
- 用户想看到的是“关系如何被击穿”，而不是“研究报告如何被展示”。

### 1.5 当前代码层面还存在未完成问题

- `plan.md` 中第二幕、第三幕部分文案尚未完整解析进 `demo_payload.json`。
- 当前 payload 中以下字段仍为空：
  - `placeholders`
  - `loading_texts`
  - `result_notes`
  - `styles.*.warm_line`
  - `styles.*.closing`
  - `styles.*.status_labels`
- 这意味着当前前端虽然能运行，但并没有完整承接最终文案方案。

## 2. 设计目标重新定义

明天修改时，建议不要把目标理解成“把页面做得更漂亮一点”，而是改成下面这句话：

**做一个关于“信息茧房如何制造心理健康误判、又如何被知识图谱击穿”的沉浸式交互前端。**

也就是说，这不是普通 demo 页面，而应该是：
- 有叙事
- 有情绪
- 有操作感
- 有反馈
- 有视觉冲击
- 有研究支撑

最终效果应该更接近：
- 互动展览
- 数字艺术化研究展示
- 可演示的实验叙事作品

而不是：
- 仪表盘
- 普通课程答辩网页
- 信息汇总页

## 3. 建议的整体视觉方向

## 3.1 关键词

- 封闭
- 噪声
- 误导
- 裂缝
- 光束
- 证据浮现
- 结构暴露
- 真假对撞

## 3.2 推荐视觉路线

建议采用“暗场 + 暖色警示 + 冷色证据光束”的方向：

- 背景主色：深灰黑、煤灰蓝、暗红棕
- 危险/误导：橙红、灼热金、脏红
- 图谱/证据：青蓝、冷白、浅电光色
- 中和层：烟灰、半透明雾层

这样能形成很明确的视觉语义：
- 错误信息 = 热、躁、诱导
- 事实证据 = 冷、锐、穿透

## 3.3 页面气质建议

- 不要再使用“规整白底卡片 + 常规圆角 + 常规表格阴影”的默认方案。
- 改为更具舞台感的层叠结构：
  - 背景雾层
  - 漂浮信息碎片
  - 中央核心交互区
  - 检索时光束 / 噪点 / 裂缝动画
  - 底部结构暴露区

## 4. 三幕式前端重设计建议

## 4.1 第一幕：茧房态

### 目标

让用户一进入页面，就感受到“被短视频话术包围”的压迫感。

### 当前问题

- 现在只是一些漂浮信息卡片，不够危险，也不够像“被算法围困”。
- 缺少沉浸感和动态层次。

### 修改建议

- 背景使用缓慢流动的暗色渐变和噪点蒙层。
- 将误导文本做成多层漂浮碎片，而不是规整卡片。
- 文本透明度不同、大小不同、漂浮速度不同。
- 部分文本可做轻微旋转、漂移、靠近、闪烁。
- 鼠标移动时，碎片产生轻微视差。
- 屏幕中心保留一个较安静的“核查入口”，形成压迫中的出口。

### 动画建议

- 页面加载时，文本不是直接出现，而是从模糊中渐显。
- 第一幕文本可持续缓慢移动，像算法流在推送。
- 当用户开始输入时，周围文本略微退散，表示“开始怀疑它”。

### 关键词文案表现

- “15秒告诉你，你可能有病”
- “中了3条以上就要小心了”
- “医生可能都没你懂自己”

这些句子应当以“攻击性投喂”的方式出现，而不是像标签展示。

---

## 4.2 第二幕：核查交互区

### 目标

把“输入一句话”升级成真正有可玩性的主交互区。

### 当前问题

- 目前只是输入框、按钮、结果卡片，太像普通表单。
- 缺少强反馈，用户操作后没有“击穿”的快感。

### 修改建议

主交互区建议做成“探照 / 扫描 / 穿透”体验：

- 输入框做成发光裂缝或扫描框，不要用普通圆角输入框。
- 按钮不叫普通按钮，而是像一个“启动核查”的装置。
- 点击后不是简单 loading，而是进入一个短暂的视觉过程：
  - 表述碎裂
  - 噪声收缩
  - 证据光束穿过
  - 最终标签落定

### 推荐交互流程

1. 用户输入一句话
2. 下方出现关键词推荐 / 相关样例推荐
3. 点击“核查”后，页面进入 0.8-1.5 秒过渡动画
4. 左侧显示原说法，右侧逐步浮现实体、关系、证据
5. 最后中心位置给出结论标签

### 推荐视觉组件

- 输入区：玻璃态 + 裂缝光边
- 推荐项：不是普通列表，而是漂浮候选片段
- 结果标签：大字冲击显示
  - “是真的”
  - “真假参半”
  - “编的”
  - “图谱未覆盖”
- 证据区：像从黑暗中打亮的证据条，而不是常规 list

### 动画建议

- 输入后候选项做渐入和上浮。
- 点击核查时，按钮收缩并触发扫描线。
- 结果标签用冲击式放大出现。
- 证据行依次错位浮现，制造“检索到东西了”的感觉。

---

## 4.3 第三幕：数据全景区

### 目标

这一幕不应该再像实验报告，而应该像“结构终于露出来了”。

### 当前问题

- 之前用了很多“图表式说明”，容易显得像答辩 PPT 嵌网页。
- 用户已经明确不希望这里堆太多关键图表。

### 修改建议

保留聚合信息，但用更清晰、更高级的可视化承载：

- 不要再放“大段解释图表”
- 不要再放“论文感标题 + 说明文字”
- 重点保留：
  - 关键统计表格
  - 对比滑块
  - 少量高辨识度结构图形

### 推荐展示方式

#### A. 关键指标改为高质量表格卡

建议保留三张信息卡，但做成更精炼的表格式：

- 子图与数据集
- 模型结果
- 幻觉与校验

要求：
- 一眼能看完
- 强调关键数字
- 弱化多余解释
- 重点数字要做层级突出

#### B. Baseline vs RAG 改为强对比滑块

这个区域仍然值得保留，因为它最有“前后差异”的戏剧性。

但视觉上应做成：
- 左边是热噪状态
- 右边是冷光校验状态
- 滑块拖动时，数字、背景、状态词同步变化

这会比单纯表格更有表现力。

#### C. 风险卡片保留

这些卡片本身很好，因为它们能说明“不是所有问题都能被图谱救回来”。

建议保留：
- 多跳推理
- 语义鸿沟
- 否定性问题
- 虚构实体

但样式要改成“告警档案卡”，不要像普通说明块。

## 5. 需要删除或弱化的内容

以下内容建议明天直接删或进一步弱化：

- 过多的解释性副标题
- 一眼看不完的长段描述
- 任何像“报告插图说明”的版块标题
- 容易被误认为作业截图的默认图卡
- 视觉存在重叠但没有功能价值的装饰元素

尤其是此前用户已经明确不想保留的“关键图表”部分，后续不要再恢复成：
- 最佳分类混淆矩阵
- Step 4 模拟纯基线图
- RAG vs 纯基线说明图

这些内容如果要体现，应被折叠进更简洁的结果表达里，而不是再次完整摆出来。

以下是当前代码层面需要继续处理的问题：

### 7.1 `plan.md` 文案解析未完成

当前文件：
- `src/primekg_mental_health/demo_final.py`

问题：
- `extract_section()` 依赖固定标题文本匹配，但 `plan.md` 中的 Markdown 结构和正则提取方式不够稳健。
- 第二幕与第三幕没有被正确截取，导致后续所有表格和列表提取失败。

受影响字段：
- `placeholders`
- `loading_texts`
- `result_notes`
- `styles.*.status_labels`
- `styles.*.warm_line`
- `styles.*.closing`
- 以及部分 `entry_prompt` / `button_label` / `panorama_intro` / `slider_left` / `slider_right`

### 7.2 Payload 与最终前端方案尚未完全同步

当前生成文件：
- `demo/demo_payload.json`
- `demo/index.html`

问题：
- 页面结构已经接近最终版，但承载的文案还没有完全换成 `plan.md` 的最终版本。
- 因此现在看到的效果只是“结构已搭好，氛围未真正成立”。

### 7.3 页面仍然偏静态

虽然已经有基础交互，但仍缺：
- 强节奏过渡
- 明确的分幕感
- 视觉冲击的状态切换
- “玩一次还想再玩一次”的反馈机制

### 7.4 语义层与视觉层没有完全绑定

理想状态应是：
- 错误说法对应热色噪声
- 核查过程对应扫描/穿透
- 证据出现对应冷色显影
- 无法覆盖对应模糊/缺口状态

当前版本仍然更多是“文本更新”，而不是“视觉语义同步变化”。

这样可以保证：
- 先把内容对齐
- 再把体验升级
- 不会在旧结构上反复修补

## 9. 推荐的最终效果判断标准

修改完成后，这个前端至少应满足以下标准：

- 用户第一眼就知道这不是普通作业网页
- 用户愿意主动输入、点击、拖动、反复尝试
- 页面有“被误导 -> 起疑 -> 击穿 -> 看清结构”的完整过程
- 中文为主，文本克制，不靠大段说明支撑
- 数据不是堆出来的，而是被设计成“有戏剧性的证据”

如果一个不看项目背景的人打开页面，会说：

“这个像一个能演示的交互作品。”

而不是：

“这是你们实验结果做成的网页。”
